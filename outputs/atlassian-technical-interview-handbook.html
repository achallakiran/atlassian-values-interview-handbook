Coding Interview Strategy

The 5-Minute Protocol Before You Write Code

  1. Clarify — Repeat the problem in your words. Ask: edge cases? input range? sorted? nulls? duplicates?
  2. Examples — Walk through 2–3 examples manually. Include an edge case.
  3. Brute force first — Name the brute force approach and its complexity. Do NOT code it yet.
  4. Optimize — State the optimized approach and why it's better. Get interviewer buy-in.
  5. Code — Only then write clean, functional Python.

The TDD Pattern Atlassian Explicitly Evaluates

Atlassian explicitly says: "Testing is crucial to success in the code task. We strongly recommend brushing up on testing, in particular the TDD approach."

Write your tests before your solution. Use this pattern in HackerRank:

def solution(input):
    # write this AFTER the tests
    pass

# Tests — write these FIRST
def test_solution():
    # happy path
    assert solution([2, 7, 11, 15], 9) == [0, 1]
    # edge: empty input
    assert solution([], 0) == []
    # edge: single element
    assert solution([5], 5) == []
    # edge: duplicates
    assert solution([3, 3], 6) == [0, 1]

test_solution()
print("All tests passed")

Multi-Solution Showcase

The guide says: "Because problems are rarely limited to a single solution, please showcase all possible solutions and why the approach you selected is the best fit."

Always present in this order:

  1. Brute force — name it, state complexity, say why it's not ideal
  2. Better approach — show the key insight (sorting, hashmap, two pointers)
  3. Optimal — implement this one, state final complexity

Communication Phrases to Use

  • "Let me start by restating the problem to make sure I understand."
  • "A brute force here would be O(n²) — let me think about a better approach."
  • "The key insight is that we can use a hashmap to get O(1) lookups."
  • "Before I code, let me write the tests first."
  • "Let me trace through this with the example input."
  • "Time complexity is O(n), space is O(n) for the hashmap."
  • "An alternative would be sorting first — that gives us O(n log n) time but O(1) space."
  • "Let me check my edge cases: empty input, single element, all same values."

Most common topic area. Focus on sliding window, frequency counting, and two-pass techniques. Python string immutability matters.

Problem 1 — Two Sum Easy

Problem: Given an array of integers nums and an integer target, return the indices of the two numbers that add up to target. Each input has exactly one solution. You may not use the same element twice.

Brute Force O(n²): Nested loop — for every pair, check if sum == target. Too slow for large inputs.
Optimal O(n): Single pass with a hashmap. For each number, check if target - num is already in the map. If yes, return indices. If no, add current num and index to map. 
def two_sum(nums: list[int], target: int) -> list[int]:
    seen = {}  # value -> index
    for i, num in enumerate(nums):
        complement = target - num
        if complement in seen:
            return [seen[complement], i]
        seen[num] = i
    return []

def test_two_sum():
    assert two_sum([2, 7, 11, 15], 9) == [0, 1]
    assert two_sum([3, 2, 4], 6) == [1, 2]
    assert two_sum([3, 3], 6) == [0, 1]
    assert two_sum([], 0) == []

test_two_sum()
print("All tests passed")
Time: O(n)
Space: O(n) — hashmap
Problem 2 — Longest Substring Without Repeating Characters Medium

Problem: Given a string s, find the length of the longest substring without repeating characters.

Brute Force O(n³): Check every substring — too slow.
Sliding Window O(n): Maintain a window [left, right]. Expand right; when we hit a duplicate, shrink from left until the duplicate is removed. Track max length. 
def length_of_longest_substring(s: str) -> int:
    char_index = {}  # char -> most recent index
    left = 0
    max_len = 0
    for right, char in enumerate(s):
        if char in char_index and char_index[char] >= left:
            left = char_index[char] + 1
        char_index[char] = right
        max_len = max(max_len, right - left + 1)
    return max_len

def test_length_of_longest_substring():
    assert length_of_longest_substring("abcabcbb") == 3
    assert length_of_longest_substring("bbbbb") == 1
    assert length_of_longest_substring("pwwkew") == 3
    assert length_of_longest_substring("") == 0
    assert length_of_longest_substring("a") == 1

test_length_of_longest_substring()
print("All tests passed")
Time: O(n)
Space: O(min(n, alphabet))
Problem 3 — Merge Intervals Medium

Problem: Given an array of intervals [[start, end], ...], merge all overlapping intervals and return the result.

Key insight: Sort by start time. Then scan: if current interval overlaps with last merged (current.start <= last.end), extend last.end. Otherwise, append current as new interval. 
def merge_intervals(intervals: list[list[int]]) -> list[list[int]]:
    if not intervals:
        return []
    intervals.sort(key=lambda x: x[0])
    merged = [intervals[0]]
    for start, end in intervals[1:]:
        if start <= merged[-1][1]:
            merged[-1][1] = max(merged[-1][1], end)
        else:
            merged.append([start, end])
    return merged

def test_merge_intervals():
    assert merge_intervals([[1,3],[2,6],[8,10],[15,18]]) == [[1,6],[8,10],[15,18]]
    assert merge_intervals([[1,4],[4,5]]) == [[1,5]]
    assert merge_intervals([[1,4],[2,3]]) == [[1,4]]
    assert merge_intervals([]) == []
    assert merge_intervals([[1,1]]) == [[1,1]]

test_merge_intervals()
print("All tests passed")
Time: O(n log n) — sort dominates
Space: O(n)
Data engineering angle: Mention this maps to session window computation in Spark Structured Streaming — you've used this concept in inventory reconciliation at Walmart.
Problem 4 — Valid Anagram Easy

Problem: Given two strings s and t, return true if t is an anagram of s.

from collections import Counter

def is_anagram(s: str, t: str) -> bool:
    if len(s) != len(t):
        return False
    return Counter(s) == Counter(t)

# Alternative without Counter — O(26) space for lowercase alpha
def is_anagram_v2(s: str, t: str) -> bool:
    if len(s) != len(t):
        return False
    counts = [0] * 26
    for c in s:
        counts[ord(c) - ord('a')] += 1
    for c in t:
        counts[ord(c) - ord('a')] -= 1
    return all(x == 0 for x in counts)

def test_is_anagram():
    assert is_anagram("anagram", "nagaram") == True
    assert is_anagram("rat", "car") == False
    assert is_anagram("", "") == True
    assert is_anagram("a", "ab") == False

test_is_anagram()

Problem 5 — Group Anagrams Medium

Problem: Given an array of strings, group the anagrams together.

Key insight: Anagrams have the same sorted characters. Use sorted string as hashmap key. 
from collections import defaultdict

def group_anagrams(strs: list[str]) -> list[list[str]]:
    groups = defaultdict(list)
    for s in strs:
        key = tuple(sorted(s))  # or ''.join(sorted(s))
        groups[key].append(s)
    return list(groups.values())

def test_group_anagrams():
    result = group_anagrams(["eat","tea","tan","ate","nat","bat"])
    result_sorted = sorted([sorted(g) for g in result])
    expected = sorted([sorted(g) for g in [["ate","eat","tea"],["nat","tan"],["bat"]]])
    assert result_sorted == expected
    assert group_anagrams([""]) == [[""]]
    assert group_anagrams(["a"]) == [["a"]]

test_group_anagrams()
print("All tests passed")
Time: O(n · k log k) where k = max string length
Space: O(n · k)
Problem 6 — Top K Frequent Elements Medium

Problem: Given an integer array and integer k, return the k most frequent elements.

Approach 1 — Sort by frequency O(n log n): Count, sort by count descending, return first k.
Approach 2 — Bucket sort O(n): Count frequencies. Create buckets indexed by frequency (max = n). Iterate buckets from high to low, collect k elements. 
from collections import Counter

def top_k_frequent(nums: list[int], k: int) -> list[int]:
    # Approach 1: heap / sort — O(n log n)
    count = Counter(nums)
    return [num for num, _ in count.most_common(k)]

def top_k_frequent_optimal(nums: list[int], k: int) -> list[int]:
    # Approach 2: bucket sort — O(n)
    count = Counter(nums)
    buckets = [[] for _ in range(len(nums) + 1)]
    for num, freq in count.items():
        buckets[freq].append(num)
    result = []
    for freq in range(len(buckets) - 1, 0, -1):
        result.extend(buckets[freq])
        if len(result) >= k:
            return result[:k]
    return result

def test_top_k_frequent():
    assert sorted(top_k_frequent([1,1,1,2,2,3], 2)) == [1, 2]
    assert top_k_frequent([1], 1) == [1]

test_top_k_frequent()
print("All tests passed")
Approach 1 Time: O(n log n)
Approach 2 Time: O(n)
Space: O(n)

Problem 7 — Container With Most Water Medium

Problem: Given an integer array height, find two lines that together with the x-axis form a container that holds the most water.

Two pointer O(n): Start left=0, right=n-1. Area = min(height[left], height[right]) * (right - left). Move the pointer with the smaller height inward. Repeat until pointers meet. 
def max_area(height: list[int]) -> int:
    left, right = 0, len(height) - 1
    max_water = 0
    while left < right:
        water = min(height[left], height[right]) * (right - left)
        max_water = max(max_water, water)
        if height[left] < height[right]:
            left += 1
        else:
            right -= 1
    return max_water

def test_max_area():
    assert max_area([1,8,6,2,5,4,8,3,7]) == 49
    assert max_area([1,1]) == 1
    assert max_area([4,3,2,1,4]) == 16
    assert max_area([1,2,1]) == 2

test_max_area()
print("All tests passed")

Space: O(1)

Stacks solve "most recent unresolved" problems. Queues (deque) solve sliding window max and BFS. Use Python's collections.deque for O(1) both ends.

Problem 9 — Valid Parentheses Easy

Problem: Given a string containing only ()[]{}, determine if the input string is valid. Open brackets must be closed in the correct order.

def is_valid(s: str) -> bool:
    stack = []
    mapping = {')': '(', '}': '{', ']': '['}
    for char in s:
        if char in mapping:
            top = stack.pop() if stack else '#'
            if mapping[char] != top:
                return False
        else:
            stack.append(char)
    return not stack

def test_is_valid():

    assert is_valid("()[]{}") == True
    assert is_valid("(]") == False
    assert is_valid("([)]") == False
    assert is_valid("{[]}") == True
    assert is_valid("") == True
    assert is_valid("]") == False

test_is_valid()

Space: O(n)
Problem 10 — Daily Temperatures Medium

Problem: Given an array of daily temperatures, return an array where each element is the number of days until a warmer temperature. If no future warmer day exists, put 0.

Monotonic stack O(n): Maintain a stack of indices with decreasing temperatures. When we find a warmer day, pop and calculate the wait. 
def daily_temperatures(temps: list[int]) -> list[int]:
    result = [0] * len(temps)
    stack = []  # indices, monotonically decreasing temps
    for i, temp in enumerate(temps):
        while stack and temp > temps[stack[-1]]:
            j = stack.pop()
            result[j] = i - j
        stack.append(i)
    return result

def test_daily_temperatures():
    assert daily_temperatures([73,74,75,71,69,72,76,73]) == [1,1,4,2,1,1,0,0]
    assert daily_temperatures([30,40,50,60]) == [1,1,1,0]
    assert daily_temperatures([30,60,90]) == [1,1,0]
    assert daily_temperatures([]) == []

test_daily_temperatures()
print("All tests passed")
Time: O(n)
Space: O(n)

Problem 11 — Level Order Traversal (BFS) Medium

Problem: Given the root of a binary tree, return the level order traversal of its nodes' values (i.e., left to right, level by level).

from collections import deque

def level_order(root) -> list[list[int]]:
    if not root:
        return []
    result = []
    queue = deque([root])
    while queue:
        level_size = len(queue)
        level = []
        for _ in range(level_size):
            node = queue.popleft()
            level.append(node.val)
            if node.left:
                queue.append(node.left)
            if node.right:
                queue.append(node.right)
        result.append(level)
    return result

def test_level_order():
    # Tree:    3
    #         / \
    #        9  20
    #           / \
    #          15   7
    root = TreeNode(3, TreeNode(9), TreeNode(20, TreeNode(15), TreeNode(7)))
    assert level_order(root) == [[3], [9, 20], [15, 7]]
    assert level_order(None) == []
    assert level_order(TreeNode(1)) == [[1]]

test_level_order()
print("All tests passed")

def max_depth(root) -> int:
    if not root:

def max_depth_iterative(root) -> int:
    if not root:

    depth = 0

Problem 13 — Validate Binary Search Tree Medium

Problem: Given the root of a binary tree, determine if it is a valid binary search tree (BST).

Key insight: Pass min/max bounds down the recursion. Left subtree: max bound = current value. Right subtree: min bound = current value. 
def is_valid_bst(root, min_val=float('-inf'), max_val=float('inf')) -> bool:
    if not root:
        return True
    if root.val <= min_val or root.val >= max_val:
        return False
    return (is_valid_bst(root.left, min_val, root.val) and
            is_valid_bst(root.right, root.val, max_val))

def test_is_valid_bst():
    # Valid BST: 2 -> (1, 3)
    root = TreeNode(2, TreeNode(1), TreeNode(3))
    assert is_valid_bst(root) == True
    # Invalid: 5 -> (1 -> (None, 4 -> (3, 6)), 4)  — 4 < 5 but is right child
    root2 = TreeNode(5, TreeNode(1), TreeNode(4, TreeNode(3), TreeNode(6)))
    assert is_valid_bst(root2) == False
    assert is_valid_bst(None) == True

test_is_valid_bst()

Problem 14 — Climbing Stairs Easy

Problem: You can climb 1 or 2 steps at a time. Given n steps, how many distinct ways can you reach the top?

Key insight: dp[i] = dp[i-1] + dp[i-2]. This is Fibonacci. Can optimize to O(1) space. 
def climb_stairs(n: int) -> int:
    if n <= 2:
        return n
    prev2, prev1 = 1, 2
    for _ in range(3, n + 1):
        prev2, prev1 = prev1, prev1 + prev2
    return prev1

Problem 15 — House Robber Medium

Problem: Houses in a row contain money. Adjacent houses have security — cannot rob two adjacent houses. Maximize the total amount robbed.

def house_robber(nums: list[int]) -> int:
    if not nums:
        return 0
    if len(nums) == 1:
        return nums[0]
    prev2, prev1 = 0, 0
    for num in nums:

    return prev1

Problem 16 — Coin Change Medium

Problem: Given coin denominations and a total amount, return the minimum number of coins to make up that amount. Return -1 if not possible.

def coin_change(coins: list[int], amount: int) -> int:
    dp = [float('inf')] * (amount + 1)
    dp[0] = 0
    for i in range(1, amount + 1):
        for coin in coins:
            if coin <= i:
                dp[i] = min(dp[i], dp[i - coin] + 1)
    return dp[amount] if dp[amount] != float('inf') else -1

def test_coin_change():
    assert coin_change([1, 5, 11], 15) == 3   # 5+5+5
    assert coin_change([2], 3) == -1
    assert coin_change([1], 0) == 0
    assert coin_change([1, 2, 5], 11) == 3    # 5+5+1

test_coin_change()
print("All tests passed")
Time: O(amount × coins)
Space: O(amount)

Pattern — Merge Sort (know the implementation) Medium

Atlassian may ask you to implement sorting. Merge sort is O(n log n) and stable.

def merge_sort(arr: list[int]) -> list[int]:
    if len(arr) <= 1:
        return arr
    mid = len(arr) // 2
    left = merge_sort(arr[:mid])
    right = merge_sort(arr[mid:])
    return merge(left, right)

def merge(left: list[int], right: list[int]) -> list[int]:
    result = []
    i = j = 0
    while i < len(left) and j < len(right):
        if left[i] <= right[j]:
            result.append(left[i]); i += 1
        else:
            result.append(right[j]); j += 1
    result.extend(left[i:])
    result.extend(right[j:])
    return result

def test_merge_sort():
    assert merge_sort([5, 2, 8, 1, 9, 3]) == [1, 2, 3, 5, 8, 9]
    assert merge_sort([]) == []
    assert merge_sort([1]) == [1]
    assert merge_sort([3, 1, 2]) == [1, 2, 3]

test_merge_sort()
print("All tests passed")
Time: O(n log n)
Space: O(n)
Problem 17 — Search in Rotated Sorted Array Medium

Problem: An integer array sorted in ascending order was rotated at an unknown pivot. Search for a target value; return its index or -1.

def search_rotated(nums: list[int], target: int) -> int:
    left, right = 0, len(nums) - 1
    while left <= right:
        mid = (left + right) // 2
        if nums[mid] == target:
            return mid
        # Left half is sorted
        if nums[left] <= nums[mid]:
            if nums[left] <= target < nums[mid]:
                right = mid - 1
            else:
                left = mid + 1
        else:  # Right half is sorted
            if nums[mid] < target <= nums[right]:
                left = mid + 1
            else:
                right = mid - 1
    return -1

def test_search_rotated():
    assert search_rotated([4,5,6,7,0,1,2], 0) == 4

String Manipulation — Clean and Parse Structured Strings Medium

Problem: Given a raw log string "2025-01-15T08:23:11Z INFO user=alice action=login status=200", parse it into a structured dict with keys: timestamp, level, user, action, status.

Approach 1 — Split + key=value parsing: Split on spaces, identify key=value pairs by checking if "=" is in the token.
Approach 2 — Regex named groups: More robust for variable-order fields. Use re.findall(r'(\w+)=(\S+)') for key-value pairs.
import re

def parse_log_line(line: str) -> dict:
    tokens = line.split()
    result = {
        'timestamp': tokens[0],
        'level': tokens[1],
    }
    for token in tokens[2:]:
        if '=' in token:
            key, _, value = token.partition('=')
            result[key] = value
    return result

def parse_log_line_regex(line: str) -> dict:
    pattern = r'^(\S+)\s+(\w+)\s+(.+)$'
    m = re.match(pattern, line)
    result = {'timestamp': m.group(1), 'level': m.group(2)}
    for k, v in re.findall(r'(\w+)=(\S+)', m.group(3)):
        result[k] = v
    return result

    r = parse_log_line(line)
    assert r['level'] == 'INFO'
    assert r['user'] == 'alice'
    assert r['status'] == '200'
    r2 = parse_log_line_regex(line)
    assert r2['action'] == 'login'
    r3 = parse_log_line("2025-01-15T08:23:11Z ERROR")
    assert r3['level'] == 'ERROR'

Dictionary Manipulation — Frequency Count, Inversion, and Merge Easy–Medium

from collections import Counter, defaultdict

# (a) Word frequency
def word_frequency(text: str) -> dict:
    return dict(Counter(text.lower().split()))

# (b) Invert dict — handle duplicate values by grouping keys into a list
def invert_dict(d: dict) -> dict:
    inverted = defaultdict(list)
    for k, v in d.items():
        inverted[v].append(k)
    return dict(inverted)

# (c) Merge two dicts, summing numeric values for shared keys
def merge_sum(d1: dict, d2: dict) -> dict:
    result = dict(d1)
    for k, v in d2.items():
        result[k] = result.get(k, 0) + v
    return result

    assert inv[2] == ['b']

    assert m == {'apac': 150, 'na': 200, 'emea': 75}

Problem — Kth Largest Element (Min-Heap) Medium

Problem: Find the kth largest element in an unsorted array without full sort.

Min-Heap O(n log k): Maintain a min-heap of size k. Push each element; if heap size exceeds k, pop the smallest. The root is the kth largest.
Quick-select O(n) avg: Partition like quicksort. Recurse only on the side containing the kth position.
import heapq

def kth_largest(nums: list[int], k: int) -> int:
    # Min-heap of size k — heap root is kth largest
    heap = []
    for num in nums:
        heapq.heappush(heap, num)
        if len(heap) > k:
            heapq.heappop(heap)
    return heap[0]

    assert kth_largest([1], 1) == 1

Problem — Merge K Sorted Lists (Min-Heap) Hard

Problem: Merge k sorted linked lists into one sorted list.

Min-Heap O(N log k): Push the head of each list into a heap. Pop minimum, push next node from that list. Repeat.
import heapq

def merge_k_sorted(lists: list[list[int]]) -> list[int]:
    # Using arrays for simplicity (adapt for ListNode)
    heap = []
    for i, lst in enumerate(lists):
        if lst:
            heapq.heappush(heap, (lst[0], i, 0))

    result = []
    while heap:
        val, list_idx, elem_idx = heapq.heappop(heap)
        result.append(val)
        next_idx = elem_idx + 1
        if next_idx < len(lists[list_idx]):
            heapq.heappush(heap, (lists[list_idx][next_idx], list_idx, next_idx))
    return result

    assert merge_k_sorted([]) == []
    assert merge_k_sorted([[1]]) == [1]

Problem — Implement Trie (Prefix Tree) Medium

Problem: Implement a Trie with insert, search, and startsWith methods.

Trie node with dict of children + end flag. Each node maps char → child node.
class TrieNode:
    def __init__(self):
        self.children = {}
        self.is_end = False

class Trie:
    def __init__(self):
        self.root = TrieNode()

    def insert(self, word: str) -> None:
        node = self.root
        for ch in word:
            if ch not in node.children:
                node.children[ch] = TrieNode()
            node = node.children[ch]
        node.is_end = True

    def search(self, word: str) -> bool:
        node = self._traverse(word)
        return node is not None and node.is_end

    def starts_with(self, prefix: str) -> bool:
        return self._traverse(prefix) is not None

    def _traverse(self, s: str):
        node = self.root
        for ch in s:
            if ch not in node.children:
                return None
            node = node.children[ch]
        return node

Problem — Number of Connected Components / Union-Find Medium

Problem: Given n nodes and a list of edges, find the number of connected components.

Union-Find (DSU): Each node starts as its own parent. Union merges components. Find with path compression returns the root. Count nodes where node == parent.
def count_components(n: int, edges: list[list[int]]) -> int:
    parent = list(range(n))
    rank = [0] * n

    def find(x):
        if parent[x] != x:
            parent[x] = find(parent[x])  # path compression
        return parent[x]

    def union(x, y):
        rx, ry = find(x), find(y)
        if rx == ry:
            return False
        if rank[rx] < rank[ry]:
            rx, ry = ry, rx
        parent[ry] = rx
        if rank[rx] == rank[ry]:
            rank[rx] += 1
        return True

    components = n
    for u, v in edges:
        if union(u, v):
            components -= 1
    return components

    assert count_components(5, [[0,1],[1,2],[2,3],[3,4]]) == 1
    assert count_components(4, []) == 4

  SELECT DISTINCT user_id, activity_date FROM activity

island_groups AS (
  SELECT
    user_id,
    activity_date,
    -- Subtract row number to get a constant "island id" per consecutive streak

    COUNT(*) AS streak_length
  FROM island_groups

HDFS Architecture & Data Locality DE Deep

Question: Explain HDFS architecture. How does data locality work, and why does it matter for Spark?

Architecture

  • NameNode: Single master — stores file system metadata (block locations, permissions). Failure = cluster failure. Mitigated by HA NameNode with ZooKeeper failover.
  • DataNodes: Store actual 128MB blocks (default). Report block health via heartbeats to NameNode every 3 seconds.
  • Replication factor 3: 1st replica on same rack node, 2nd on different rack, 3rd on different rack. Tolerates one full rack failure.
  • Secondary NameNode: NOT a hot standby — it periodically merges the edit log with the fsimage to prevent edit log growth. Common misconception in interviews.

Data Locality

  • NODE_LOCAL: Task runs on the node where the data block lives. Zero network I/O. Best case.
  • RACK_LOCAL: Task runs on a node in the same rack. Intra-rack bandwidth (~1Gbps). Acceptable.
  • ANY: Data must be transferred over inter-rack network. Slowest — avoided by the YARN scheduler.
Spark connection: Spark's DAGScheduler tries NODE_LOCAL first, waits for a configurable locality wait time (spark.locality.wait), then falls back to RACK_LOCAL, then ANY. On GCP Dataproc, HDFS is replaced by GCS — there's no data locality. Google compensates with high-bandwidth internal network, but this is a key architectural difference you should name.
GCP migration insight: When you moved from on-prem HDFS to GCS, the data-locality advantage disappeared. You compensated by using Dataproc autoscaling, pre-warming clusters, and tuning spark.executor.cores and spark.default.parallelism to maximize CPU utilization over the high-bandwidth GCS network.
MapReduce — Internals, Shuffle, and When NOT to Use It DE Deep

Question: Explain MapReduce internals. When would Spark outperform MapReduce and when might MapReduce still be appropriate?

MapReduce Internals

  1. Map phase: Each mapper reads input splits. Applies user function. Outputs (key, value) pairs. Output is partitioned by key using the partitioner (default: hash(key) % reducers).
  2. Shuffle & Sort: The bottleneck. Framework sorts map outputs by key on disk, then transfers them over the network to the correct reducer. This is disk I/O intensive.
  3. Reduce phase: Each reducer receives all values for its key range in sorted order. Applies reduce function. Writes to HDFS.
  4. Combiner (optional): A mini-reducer that runs on the mapper side before shuffle to reduce network traffic.

Lambda vs Kappa Architecture DE Deep

Question: When would you choose Lambda architecture over Kappa, and vice versa? What are the operational tradeoffs?

Lambda Architecture

  • Batch layer: Reprocesses all historical data on schedule. Source of truth. High latency (hours).
  • Speed layer: Processes real-time stream for low latency. Approximate or incomplete.
  • Serving layer: Merges batch and speed layer outputs to serve queries.
  • Problem: Two codebases to maintain — batch logic and streaming logic must produce identical results. Debugging drift between layers is painful.

Kappa Architecture

  • Single streaming pipeline handles all data. Reprocess historical data by replaying the event log (Kafka retention).
  • Advantage: One codebase, one processing model. Simpler ops.
  • Problem: Reprocessing large history through streaming is slow. Kafka must retain very long history. Streaming frameworks are harder to tune for complex batch-style aggregations.
Decision framework: Choose Lambda when: (1) batch correctness matters more than operational simplicity, (2) reprocessing is rare, (3) the speed layer only needs approximate results. Choose Kappa when: (1) the team can maintain only one codebase, (2) Kafka can retain enough history, (3) your streaming framework (Flink) handles batch semantics well. At Walmart, your fraud detection system was effectively Kappa — Kafka + Spark Structured Streaming was the single path, with replay capability for model retraining.

Partitioning Strategies — How to Partition a 10TB Table DE Deep

CAP Theorem — Practical Application DS Deep

Question: Explain the CAP theorem. When would you choose CP over AP for a data engineering system? Give a concrete example.

CAP Theorem

  • Consistency (C): Every read receives the most recent write or an error. All nodes see the same data at the same time.
  • Availability (A): Every request receives a (non-error) response — though it may not contain the most recent write.
  • Partition Tolerance (P): The system continues to operate even if network messages between nodes are dropped or delayed.
The nuance: Partition tolerance is not optional in distributed systems — network partitions happen. The real choice is: when a partition occurs, do you prioritize consistency (CP) or availability (AP)?

Consistency Models — Strong, Eventual, Causal DS Deep

Question: Explain strong consistency, eventual consistency, and causal consistency. Where does each appear in a data platform?

  • Strong consistency: All reads reflect the most recent completed write. Requires coordination (locks, 2PC, Paxos/Raft). Latency penalty. Use: financial ledgers, metadata stores (Hive Metastore), config systems.
  • Eventual consistency: Given no new writes, all replicas will converge to the same value — eventually. No coordination overhead. Use: DNS, shopping carts, analytics event stores, Cassandra default, DynamoDB default.
  • Causal consistency: Operations that are causally related are seen in causal order by all nodes. Writes that are not causally related may be seen in different orders. Stronger than eventual, weaker than strong. Use: social media timelines (a reply must appear after its parent post), distributed collaboration tools (like Confluence page edits).
  • Read-your-writes consistency: After you write, you always read your own write. Critical for user-facing apps — a user who updates a Jira ticket must see their own update immediately.

Partitioning Strategies — Hash, Range, Directory DS Deep

Question: How does Kafka partition data? How does Cassandra? What problems does each solve, and what anti-patterns do they create?

Hash Partitioning

  • Assign partition by hash(key) % num_partitions. Even data distribution. No range queries possible across partitions — must query all partitions.
  • Kafka: Default behavior — messages with the same key go to the same partition, preserving per-key ordering. No key → round-robin across partitions.
  • Cassandra: Uses consistent hashing (Murmur3 hash of partition key maps to a token ring position). Enables adding nodes without full re-shuffle.
  • Hot partition problem: If 80% of traffic is for one user (e.g., a celebrity account), that partition is hot. Solution: add a random suffix to the key (salt) and query all salted partitions — trades query complexity for even distribution.

Range Partitioning

  • Partitions own a contiguous range of key values. Supports efficient range scans. Risk: monotonically increasing keys (timestamps, IDs) cause the last partition to always be the write hotspot.
  • Mitigation: Reverse the timestamp bytes, add randomness to the key, or use write-distribution strategies like Kafka key salting.

Distributed Transactions — 2PC, Saga, Outbox Pattern DS Deep

Question: How do you ensure a data pipeline that writes to both BigQuery and Kafka stays consistent without distributed transactions?

2-Phase Commit (2PC)

  • Phase 1 — Prepare: coordinator asks all participants to lock resources and vote yes/no.
  • Phase 2 — Commit/Abort: if all vote yes, coordinator sends commit. If any vote no, abort.
  • Problems: Blocking — if coordinator fails after phase 1, participants hold locks indefinitely. Low throughput due to synchronous coordination. Not practical at data engineering scale.

Saga Pattern

  • Break a distributed transaction into a sequence of local transactions. Each step publishes an event/message to trigger the next. If a step fails, compensating transactions undo prior steps.
  • Choreography: Each service listens for events and acts autonomously. Simple but hard to track state.
  • Orchestration: A central saga orchestrator sends commands to each service. Easier to track but single point of failure.

Transactional Outbox Pattern

# Problem: write to DB and publish to Kafka atomically
# Solution: write to an outbox table in the same DB transaction,
# then a separate process reads the outbox and publishes to Kafka.

  INSERT INTO orders(id, user_id, amount) VALUES (...);
  INSERT INTO outbox(event_type, payload, status)
    VALUES ('ORDER_CREATED', '{"id":...}', 'PENDING');
COMMIT;

Replication — Leader-Follower vs Multi-Leader vs Leaderless DS Deep

Question: Explain the three replication topologies. What are the failure scenarios and how do you handle them?

  • Leader-Follower (Primary-Replica): All writes go to leader. Leader replicates to followers. Followers serve reads. On leader failure, a follower is promoted (automatic with tools like Patroni for Postgres, Kafka controller election). Risk: replication lag — reads from followers may be stale.
  • Multi-Leader: Multiple leaders accept writes. Each leader replicates to the others. Used in multi-datacenter setups. Problem: write conflicts when same record is written to two leaders concurrently. Must resolve conflicts (last-write-wins, CRDTs, application-level merge). Used by Cassandra, CouchDB, Atlassian Confluence's offline mode.
  • Leaderless (Dynamo-style): Any node accepts writes. Reads and writes use quorum (W + R > N). Example: Cassandra, DynamoDB. Read-repair and anti-entropy daemons keep replicas in sync. No single point of failure but complex consistency guarantees.
Quorum: W + R > N guarantees overlap
Strong: W = N (write all replicas)
Fast: W = 1 (write one, eventual)

PySpark Internals — DAG, Stages, Shuffles, and Optimization Frameworks

Question: A Spark job is slow. Walk me through how you would diagnose and fix it.

Spark Execution Model

  1. Driver submits a job. Spark builds a DAG of RDD/DataFrame transformations.
  2. DAG Scheduler divides the DAG into stages — a new stage begins at every shuffle boundary (wide transformation: groupBy, join, distinct).
  3. Each stage is split into tasks (one per partition). Tasks run on Executor cores.
  4. Shuffle = data redistributed across the network between stages. The most expensive operation.

Diagnosis Checklist (Spark UI)

  • Jobs tab: Which job is slow? Which stage inside it?
  • Stages tab: Is one task taking 10x longer than others? → Data skew.
  • Storage tab: Is the cached RDD fitting in memory? Spill to disk = slow.
  • SQL/DataFrame tab: Look for BroadcastHashJoin vs SortMergeJoin — BHJ is faster when one side fits in memory.
  • Executor tab: GC time >20% of task time = memory pressure, need to increase executor memory or reduce partition size.

Kafka Architecture — Producers, Consumers, Partitions, and Delivery Guarantees Frameworks

Question: Explain Kafka's architecture. How do you achieve exactly-once delivery end-to-end?

Core Architecture

  • Topic: Named log of records. Divided into partitions. Partitions are the unit of parallelism.
  • Partition: Immutable, ordered, append-only log. Each record has an offset. Replicated across brokers.
  • Producer: Writes to a partition. Key → deterministic partition (same key → same partition, ordering guaranteed). No key → round-robin.
  • Consumer Group: A group of consumers that collectively consume a topic. Each partition is consumed by exactly one consumer in the group at a time. Max parallelism = number of partitions.
  • Controller (KRaft mode): Manages partition leadership and metadata. ZooKeeper removed in Kafka 3.3+ (KRaft mode is now default).

Airflow — DAG Design, Idempotency, and Common Anti-Patterns Frameworks

Question: How do you design an Airflow DAG for a daily batch pipeline that processes yesterday's data? What makes it production-ready?

Production-Ready DAG Checklist

  • Idempotency: Re-running the same DAG run for the same ds (execution date) must produce the same result. Use execution_date as a partition key. Delete before insert, not append.
  • Catchup disabled by default: Set catchup=False unless you specifically need backfill. An unintended catchup on a 30-day DAG runs 30 concurrent DAG runs and kills your database.
  • SLA misses: Set sla on tasks to get alerts when a task takes longer than expected.
  • Retries + backoff: retries=3, retry_delay=timedelta(minutes=5). Use exponential backoff for flaky external APIs.
  • XComs only for small metadata: Never pass DataFrames through XComs. Pass file paths, BigQuery table names, or row counts only.
  • Use datasets/data-aware scheduling (Airflow 2.4+) to trigger downstream DAGs when upstream data is ready — decouples scheduling from clock time.
